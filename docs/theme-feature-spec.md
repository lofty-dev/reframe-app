# 実装指示書：「次回までのテーマ」機能

Stride に「次回までのテーマ」機能を追加する。診察の最後に主治医・支援者と決めた「次回までに見ておくこと」を1行で記録し、次の診察までの参照軸にする機能。

## 背景と設計思想（実装判断に迷ったらここに立ち返る）

- Strideは診察の入口（伝えたいことメモ）とその場（Bridge Session)を支援しているが、出口（診察で決まったこと）が構造化されていない。そこを埋める。
- **独立した新機能・新画面は作らない**。既存の診察関連フローに1フィールドと表示箇所を足すだけの最小実装とする。
- テーマは「ユーザーの目標」ではなく「支援者と決めた観察の軸」。ゆえに支援者ごとに管理し、診察の文脈でのみ表示する。
- **ホーム・チェックイン画面には一切表示しない**。日常の記録場面でテーマを見せると記録内容がテーマに誘導され、データの自然さが失われるため。これは意図的な仕様であり、「せっかくだからホームにも」としないこと。

---

## 1. データモデル

localStorage に新キー `stride_themes` を追加。値はテーマオブジェクトの配列。

```json
{
  "id": "theme_<timestamp>_<random>",
  "supporterId": "既存の支援者管理と同じID",
  "text": "睡眠時間の変化を見ていく",
  "createdAt": "2026-07-03T10:00:00.000Z",
  "sourceRecordId": "作成契機となった診察記録のID。手動作成時はnull",
  "status": "active",
  "closedAt": null,
  "closedRecordId": null
}
```

- `status` は `"active" | "closed"` の2値のみ。
- `text` は最大100文字。入力UIでカウンター表示し、超過時は保存不可。
- **制約：同一 `supporterId` に対して `active` なテーマは常に最大1件。** この制約は保存処理の関数内で強制する（新規activeを保存する際、既存activeがあれば先にclosedへ更新する。UI側の分岐に依存させない）。

`storage.js` に読み書きヘルパーを追加：`getThemes()` / `saveThemes(themes)`、および `getActiveTheme(supporterId)`、`createTheme({supporterId, text, sourceRecordId})`、`closeTheme(themeId, closedRecordId)` 相当の操作関数。既存ヘルパーの命名規則・エラーハンドリング方針に合わせること。

### 既存機能への影響

- JSONエクスポート／インポートの対象キーに `stride_themes` を追加する。エクスポート処理がキーを列挙している場合は追加漏れに注意。インポート時、`stride_themes` が存在しない旧バックアップでもエラーにしない。
- 支援者が削除された場合の挙動は、**既存の「診察等の記録」が支援者削除時にどう扱われているかを確認し、それと同じ方針に揃える**（連動削除なら連動削除、保持なら保持）。独自方針を新設しない。
- マイグレーション不要。既存データには触らない。

## 2. 入力導線（3箇所）

テーマ入力は共通コンポーネント（仮称 `ThemePrompt`）として実装し、3箇所から呼び出す。

### 2-1. 伝えたいことメモの全チェック完了時

既存フロー「全項目チェック → 完了 → 診察記録に自動反映」の**最後**に1画面追加する。

- 見出し：「次回までのテーマはありますか？」
- 補足文：「診察で決まった『次回までに見ておくこと』があれば記録できます」
- プレースホルダー：「例：睡眠時間を見ていく／頓服を使ったタイミングをメモする」
- ボタン：「保存する」「スキップ」。**スキップを常に等価な選択肢として表示**し、未入力を咎めるトーンにしない。
- この支援者にすでにactiveなテーマがある場合の分岐は「4. ライフサイクル」参照。
- 保存時、`sourceRecordId` に今回自動生成された診察記録のIDを紐付ける。

### 2-2. Bridge Session 終了時

セッション終了画面（既存）に同じ `ThemePrompt` を組み込む。挙動は2-1と同一。Bridge Sessionは開始時に支援者を選択して紐付く設計（登録が1人の場合は選択スキップ）のため、テーマの `supporterId` にはそのセッションの支援者をそのまま使う。追加の支援者選択UIは不要。

### 2-3. 診察等の記録画面から手動追加・編集

伝えたいことメモやBridge Sessionを経由しない診察もあるため、手動導線を必ず用意する。

- 診察等の記録の支援者タブ内、テーマ表示カード（後述3-1）から「追加」または「編集」できる。
- 手動作成時は `sourceRecordId: null`。
- 編集はactiveなテーマの `text` の修正のみ可能（誤字直しを想定）。closedなテーマは編集不可。

## 3. 表示箇所（3箇所）

### 3-1. 診察等の記録：支援者タブ上部（常設）

支援者タブを開いたとき、記録一覧の上に「いま見ているテーマ」カードを常設表示する。

- activeなテーマがある場合：ラベル「次回までのテーマ」＋本文＋設定日（「6/12の診察で設定」の形式。`sourceRecordId` がnullなら日付のみ）。
- ない場合：控えめな空状態「次回までのテーマは未設定です ＋追加」。空状態を大きく目立たせない。
- 過去のテーマ（closed）は、カード下の折りたたみ「過去のテーマ」で時系列表示する。表示内容：本文＋期間（createdAt〜closedAt）。この履歴は「この数ヶ月、支援者と何を見てきたか」の記録として価値があるため削除機能は初版では不要（編集も不可）。

### 3-2. 伝えたいことメモ：新規作成時の参照表示

新しい伝えたいことメモを作成する画面で、宛先の支援者にactiveなテーマがあれば、入力欄の上に参照表示する。

- 形式：「前回からのテーマ：睡眠時間の変化を見ていく」の1行。タップ等のアクションは不要、読むだけの表示。
- 目的はメモを書く観点の想起。テーマをメモ項目に自動変換するような機能は**作らない**（テーマは話題の候補ではなく観察の軸であり、毎回話題化を強制するものではない）。

### 3-3. Bridge Session：表示パネル

Bridge Sessionの表示項目カスタマイズに「前回からのテーマ」を追加する。

- デフォルト：ON（機微度が低く、診察冒頭の導入として機能するため）。
- 表示形式は3-1のカードに準ずる。セッション対象の支援者のactiveテーマのみ表示。
- パネル設定の永続化は既存のBridge Session設定の保存方式に合わせる。

## 4. ライフサイクル

### 新規テーマ入力時に既存のactiveテーマがある場合

`ThemePrompt` は先に既存テーマを提示し、3択にする：

1. **「このテーマを続ける」**：何も変更せず閉じる（既存テーマがactiveのまま残る）。
2. **「完了して新しいテーマを設定」**：既存を `closed` にし（`closedAt` と、文脈があれば `closedRecordId` を記録）、新規入力欄を表示。
3. **「完了する（新しいテーマなし）」**：既存を `closed` にして閉じる。

「続ける」に状態変更は伴わない（キャリーオーバーを表すステータスは設けない）。

### 完了の意味づけ

closedは「達成」ではなく「観察期間の区切り」。UI文言に「達成」「クリア」等の評価的な語を使わないこと。「完了」または「区切る」を使う。

---

## 5. 文言・トーン規定

- 機能名はUI上一貫して**「次回までのテーマ」**。「目標」「宿題」「課題」という語は使わない（評価・義務のニュアンスを避ける）。
- テーマ未設定・スキップを負として扱う文言（「まだ設定されていません！」等）を使わない。テーマがない診察は普通にある。
- 通知・リマインダーは実装しない。

---

## 6. 実装上の注意

- App.jsx は単一ファイル・view state による画面遷移で構成されている。既存の view 命名規則、ページコンポーネントの分割粒度、スタイル（styles.js）の流用方針に従うこと。
- `ThemePrompt` は components/ 配下に切り出してよい（3箇所から使うため）。
- 定数文言（プレースホルダー例文など）は constants.js に置く。
- ダーク／ライト両テーマで表示確認すること。
